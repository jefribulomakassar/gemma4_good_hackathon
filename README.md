## 🌍 Problem Statement

Indonesia has **1 doctor per 5,000 people** — far below WHO's recommended 1:600 ratio.  
In 3T regions (Tertinggal, Terdepan, Terluar), over **45% of Puskesmas lack adequate medical staff**.  
Community health workers (kaders) — often non-medical volunteers — are the **only frontline health resource** for millions of people across remote islands, highlands, and coastal villages.

These kaders must make triage decisions **without internet, without doctors nearby, and without structured guidance.**

**PuskesmasAI solves this.**

---

## 💡 Solution

A **mobile-first Progressive Web App (PWA)** that runs **fully offline** using Gemma 4 (E4B quantized),  
enabling kaders to:

- Input patient symptoms via simple form or voice (Bahasa Indonesia + local dialects)
- Receive structured AI triage: GREEN / YELLOW / ORANGE / RED
- Get actionable referral guidance with auto-generated referral notes
- Log patient records locally — syncs to Puskesmas dashboard when connectivity returns
- Access offline medical reference library (top 10 rural diseases, drug dosages, emergency protocols)

**No internet required after first setup. Works on low-end Android via browser.**

## 🌐 What is a PWA? (Beginner Guide)

> **If you have never built a PWA before, this section explains everything from scratch.**

A **Progressive Web App** is a regular website that behaves like a native mobile app — no app store required.

| Feature | Regular Website | Native App (Flutter) | PWA |
|---|---|---|---|
| Works offline | ❌ | ✅ | ✅ |
| Install to home screen | ❌ | ✅ (via store) | ✅ (via browser) |
| App store required | — | ✅ | ❌ |
| Instant updates | ✅ | ❌ (store review) | ✅ |
| APK file to extract/attack | — | ✅ | ❌ |
| Built with | HTML/JS | Dart | HTML/JS (same as web) |

### The 3 building blocks of a PWA

#### Block 1 — `manifest.json`
Tells the browser: *"this website is also an installable app."*

```json
{
  "name": "PuskesmasAI",
  "short_name": "PuskesmasAI",
  "description": "Offline AI triage assistant for Puskesmas health workers",
  "start_url": "/",
  "display": "standalone",
  "background_color": "#0f4c35",
  "theme_color": "#16a34a",
  "icons": [
    { "src": "/icons/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "/icons/icon-512.png", "sizes": "512x512", "type": "image/png" }
  ]
}
```

Place this file at `frontend/public/manifest.json`.  
Add this line to your `<head>` in `layout.tsx`:
```html
<link rel="manifest" href="/manifest.json" />
```

#### Block 2 — Service Worker
A background script that intercepts every network request and decides: *"serve from cache or fetch from network?"*

With **next-pwa**, you do NOT write the service worker manually. It is auto-generated:

```bash
npm install next-pwa
```

```js
// frontend/next.config.js
const withPWA = require('next-pwa')({
  dest: 'public',           // sw.js is placed here automatically
  register: true,           // auto-registers on page load
  skipWaiting: true,        // new SW activates immediately on update
  disable: process.env.NODE_ENV === 'development',

  runtimeCaching: [
    {
      // Triage API: try network first, fall back to cached response
      urlPattern: /^https:\/\/.*\/api\/triage/,
      handler: 'NetworkFirst',
      options: {
        cacheName: 'api-triage-cache',
        expiration: { maxEntries: 50, maxAgeSeconds: 3600 }
      }
    },
    {
      // Medical reference: cache-first (rarely changes)
      urlPattern: /^https:\/\/.*\/api\/reference/,
      handler: 'CacheFirst',
      options: {
        cacheName: 'reference-cache',
        expiration: { maxEntries: 10, maxAgeSeconds: 604800 } // 7 days
      }
    },
    {
      // All pages and static assets
      urlPattern: /^https?.*/,
      handler: 'NetworkFirst',
      options: { cacheName: 'page-cache' }
    }
  ]
})

module.exports = withPWA({ reactStrictMode: true })
```

#### Block 3 — HTTPS
PWA Service Workers only activate over HTTPS.
- **Development**: `localhost` is treated as secure by Chrome ✅
- **Production**: Vercel provides free HTTPS automatically ✅

No extra configuration needed.

### How offline works — step by step

```
FIRST VISIT (online):
  1. User opens https://puskesmasai.vercel.app in Chrome
  2. Service Worker installs and caches:
     - All HTML, CSS, JS files
     - Icons and fonts
     - Medical reference data (from /api/reference)
  3. Gemma 4 is already running on the local Flask server

SUBSEQUENT VISITS (no internet):
  1. Chrome opens the app from Service Worker cache
  2. Triage form works normally — user fills in symptoms
  3. Form submits to Flask on localhost:5000 (always available)
  4. Gemma 4 processes the request locally
  5. Result displayed — patient record saved to IndexedDB

WHEN INTERNET RETURNS:
  1. navigator.onLine becomes true
  2. sync.ts automatically uploads pending IndexedDB records to Turso
  3. New medical_kb.json is fetched and cached for future offline use
```

### Offline patient data storage — IndexedDB with Dexie.js

IndexedDB is a built-in browser database. Dexie.js makes it easy to use:

```bash
npm install dexie
```

```ts
// frontend/src/lib/db.ts
import Dexie, { Table } from 'dexie'

export interface PatientRecord {
  id?: number
  kader_id: string
  patient_age: number
  patient_sex: string
  symptoms: string
  triage_level: string
  recommendation: string
  timestamp: number
  synced: boolean        // false until uploaded to Turso
}

class PuskesmasDB extends Dexie {
  patients!: Table<PatientRecord>

  constructor() {
    super('PuskesmasAI')
    this.version(1).stores({
      patients: '++id, kader_id, timestamp, synced'
    })
  }
}

export const db = new PuskesmasDB()

// Save a triage result locally
export async function savePatientRecord(record: Omit<PatientRecord, 'id'>) {
  return await db.patients.add({ ...record, synced: false })
}

// Get all unsynced records (to upload when online)
export async function getUnsyncedRecords() {
  return await db.patients.where('synced').equals(0).toArray()
}
```

## 🔐 Backend Security Architecture

> **Core principle**: The backend is the only layer you fully control.  
> Whether the client is PWA, Flutter, or React Native — a determined attacker can always inspect client code.  
> **All security logic must live in the backend.**

### Why PWA has a smaller attack surface than Flutter

<table>
<tr><th>Risk</th><th>Flutter APK</th><th>PWA</th></tr>
<tr><td>File to extract</td><td>APK downloadable, decompilable with jadx/apktool</td><td>No installable binary exists</td></tr>
<tr><td>String secrets</td><td>Readable even after obfuscation</td><td>Secrets never reach client</td></tr>
<tr><td>Patch a vulnerability</td><td>Rebuild + Play Store review (days/weeks)</td><td>Redeploy on Vercel (seconds)</td></tr>
<tr><td>Obfuscation</td><td>Partial (renames symbols, not values)</td><td>N/A — source minified server-side</td></tr>
</table>

**But the same 6 backend defense layers protect you regardless of client technology:**

---

### Layer 1 — JWT Authentication
```python
# backend/middleware/auth.py
import jwt
from functools import wraps
from flask import request, jsonify

SECRET = os.environ['JWT_SECRET']   # never hardcoded

def require_auth(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        token = request.headers.get('Authorization', '').replace('Bearer ', '')
        try:
            payload = jwt.decode(token, SECRET, algorithms=['HS256'])
            request.kader_id = payload['kader_id']
        except jwt.ExpiredSignatureError:
            return jsonify({'error': 'Token expired'}), 401
        except jwt.InvalidTokenError:
            return jsonify({'error': 'Invalid token'}), 401
        return f(*args, **kwargs)
    return decorated
```

Kaders log in once → receive a JWT valid for 8 hours → every triage request carries this token.

---

### Layer 2 — HMAC Request Signing
```python
# backend/middleware/hmac_verify.py
import hmac, hashlib, os

SHARED_SECRET = os.environ['HMAC_SECRET'].encode()

def verify_hmac(request_body: bytes, signature: str) -> bool:
    expected = hmac.new(SHARED_SECRET, request_body, hashlib.sha256).hexdigest()
    return hmac.compare_digest(expected, signature)
```

Every request body is signed on the client side. The backend rejects anything with a wrong signature — **blocks forged or replayed requests** from unauthorized clients.

---

### Layer 3 — Rate Limiting
```python
# backend/middleware/rate_limit.py
from flask_limiter import Limiter
from flask_limiter.util import get_remote_address

limiter = Limiter(
    key_func=get_remote_address,
    default_limits=["200 per day", "30 per minute"]
)

# Applied to triage endpoint specifically
@app.route('/api/triage', methods=['POST'])
@limiter.limit("5 per 10 seconds")   # burst protection
@require_auth
def triage():
    ...
```

Even if a token is stolen, the attacker cannot flood the endpoint.

---

### Layer 4 — Strict CORS
```python
# backend/middleware/cors.py
from flask_cors import CORS

# Only requests from the official PWA domain are accepted
CORS(app, origins=["https://puskesmasai.vercel.app"])
```

Requests from unknown origins (e.g., Postman with a spoofed client, or a forked app) are blocked at the HTTP level.

---

### Layer 5 — Input Validation + Prompt Injection Guard
```python
# backend/services/prompt_builder.py

BLOCKED_PATTERNS = [
    "ignore previous instructions",
    "you are now",
    "system:",
    "assistant:",
    "<|im_start|>"
]

def build_triage_prompt(data: dict) -> str:
    symptoms = data.get('symptoms', '')

    # Strip prompt injection attempts
    for pattern in BLOCKED_PATTERNS:
        if pattern.lower() in symptoms.lower():
            symptoms = "[SANITIZED INPUT]"
            break

    # Truncate to safe length
    symptoms = symptoms[:500]

    return f"""You are a clinical triage assistant for Indonesian community health workers.
A kader reports the following patient:
- Age: {data['age']} years old
- Sex: {data['sex']}
- Symptoms: {symptoms}
- Temperature: {data.get('temp', 'not recorded')} °C

Respond ONLY with valid JSON in this exact format:
{{
  "triage_level": "GREEN|YELLOW|ORANGE|RED",
  "recommendation": "...",
  "possible_conditions": ["..."],
  "immediate_actions": ["..."],
  "red_flags": ["..."],
  "confidence": 0.0-1.0,
  "disclaimer": "This is a decision-support tool only."
}}"""
```

---

### Layer 6 — No Patient Data in Server Logs
```python
# backend/app.py

import logging

# Custom log filter — strips any field that might contain patient info
class PatientDataFilter(logging.Filter):
    SENSITIVE_KEYS = {'symptoms', 'patient_name', 'age', 'sex'}

    def filter(self, record):
        if hasattr(record, 'request_body'):
            for key in self.SENSITIVE_KEYS:
                record.request_body.pop(key, None)
        return True

app.logger.addFilter(PatientDataFilter())
```

Server logs only contain: timestamp, triage_level, kader_id (anonymized hash), response_time. No symptoms or patient identifiers ever persisted in logs.

## 🧠 How Gemma 4 Is Used

### 1. Structured Triage Reasoning
Gemma 4 returns strict JSON — not free-form text. The response is schema-validated before being sent to the client.

**Example:**
```
Input: "32yo female, demam 3 hari, bintik merah, mual, temp 38.8°C"

Output:
{
  "triage_level": "ORANGE",
  "recommendation": "Refer to Puskesmas immediately",
  "possible_conditions": ["Dengue Fever", "Typhoid Fever"],
  "immediate_actions": [
    "Monitor temperature every 2 hours",
    "Oral rehydration — ORS if available",
    "Do NOT give ibuprofen or aspirin (dengue risk)"
  ],
  "red_flags": [
    "Bleeding from gums or nose → RED, emergency referral",
    "Platelet drop signs → immediate hospital"
  ],
  "confidence": 0.81
}
```

### 2. Local Language Understanding
Gemma 4 understands natural Bahasa Indonesia layperson descriptions:
- "pusing muter-muter" → vertigo / rotational dizziness
- "badan panas dingin" → fever with alternating chills
- "perut kembung dan mulas" → abdominal bloating + cramping

### 3. On-Device Inference — No Cloud
Gemma 4 E4B GGUF quantized (~2.5GB) runs via Ollama locally.  
**Zero patient data ever leaves the community.** Privacy-by-design.
